# 📋 오늘 작업 정리 — 암묵지 후보 생성 전처리 파이프라인

> 처음 보는 사람을 위한 정리. "이 파이프라인이 무엇을, 어떤 모델로, 어떻게 하는가"를 이번 실제 실행(CLIP3) 결과와 함께 설명한다.
> 작성: 2026-06-30. 실행/환경 함정의 상세는 `_실행런타임_보고서.md`, 설계 배경은 `_상황설명_READ_FIRST.md` 참조(여기선 중복 안 함).
> ⚠️ 이 문서는 **현재까지 확인된 것**만 적었다. 추가 결과가 나오면 갱신한다.

## 1. 전체 그림

숙련자(명장)가 스마트글래스(머리에 쓰는 카메라 안경)로 찍은 컴퓨터 수리 영상에서, **행동(영상)**과 **설명(말소리)**을 따로 뽑아 정제한 뒤, 마지막에 LLM(대규모 언어모델)이 둘을 합쳐 **"암묵지 후보"(말로 잘 안 적히는 숙련 노하우)를 구조화된 JSON으로** 만든다. 우리 범위는 **후보 생성까지** — 진짜 암묵지인지 판별은 다음 팀(품질검증) 몫.

- **오디오(음성) 파이프라인**: 영상의 소리 → STT(음성→텍스트 변환)로 발화를 받아쓰고, 가벼운 정제(오타 교정·환각 태깅)를 한다.
- **비디오(영상) 파이프라인**: 영상의 화면 → 프레임(정지화면)을 뽑아 → YOLO(물체 위치 검출)로 부품을 찾고 → VLM(영상을 보고 글로 설명하는 모델)로 "무슨 행동을 하는지"를 관찰 기록한다.
- **합쳐지는 곳**: 두 갈래는 끝까지 따로 가다가 **타임스탬프(영상 0:00 기준 시각)**를 접착제로 **LLM 융합 단계**에서 처음 만난다. 시각이 비슷한 행동·발화를 묶어 후보 JSON 1건으로 만든다.

```
스마트글래스 영상 (화면 + 소리)
   │
   ├─ [오디오] ───────────────┐        ┌──── [비디오] ─────────────────┐
   │  소리                     │        │  화면                          │
   │   ↓ STT (Whisper)         │        │   ↓ 프레임 추출 (ffmpeg)        │
   │  발화 텍스트(+시각)        │        │   ↓ YOLO (부품 위치 검출)       │
   │   ↓ 정제(정규화·환각태깅)  │        │   ↓ VLM (행동을 글로 관찰)      │
   │  정제 transcript          │        │  관찰 로그(+시각)               │
   └───────────────┬──────────┘────────┴───────────┬───────────────────┘
                   │   (타임스탬프 = 영상 0:00 기준 공통 시각으로 정렬)
                   ↓
            LLM 융합 (Qwen2.5, 시각으로 행동+말 연결)
                   ↓
            암묵지 후보 JSON (스키마 v1.3)  →  (이후 품질검증/DB/RAG: 우리 범위 밖)
```

## 2. 오디오 파이프라인 상세

- **모델/도구**: `faster-whisper` 백엔드의 **Whisper large-v3-turbo** (정확한 식별자: `mobiuslabsgmbh/faster-whisper-large-v3-turbo`). Whisper = OpenAI의 음성인식 모델, large-v3-turbo = 빠른 경량판.
- **주요 설정**: 언어 `ko`(한국어), `condition_on_previous_text=False`(앞 문장 맥락을 다음 인식에 안 넘김 → 헛소리·반복 환각 억제).
- **입력 → 처리 → 출력**:
  - 입력: 영상 파일(소리 트랙).
  - 처리: 발화를 segment(말 토막) 단위로 받아쓰고 각 토막에 시작/끝 시각(초)을 붙임 → 이어서 **정제**(기술용어 정규화 예: "마더보이드"→"motherboard", 끝부분 반복 환각 태깅). 정제는 **삭제 안 하고 태깅만** 하고, 근거성 판단은 안 함(그건 융합 LLM 몫).
  - 출력: `transcripts/<video_id>.json` (발화 목록 + 시각 + 정규화/환각 플래그).
- **이번 런 STT 원시 산출물** (경로: `transcripts/CLIP3_재부팅시도 전까지.mp4.json`) — 발췌:
  ```json
  { "model": "mobiuslabsgmbh/faster-whisper-large-v3-turbo", "language": "ko",
    "utterances": [
      {"start": 0.0,  "end": 1.42,  "raw_text": "시작합니다", "repeat_hallucination": false},
      {"start": 30.0, "end": 32.0,  "raw_text": "아멘", "repeat_hallucination": false},
      {"start": 60.0, "end": 64.86, "raw_text": "깨끗해 보여도 이렇게 한 번씩 닦아줘야 합니다."},
      {"start": 65.64,"end": 68.02, "raw_text": "산화막이 접촉불량을 만들 수가 있습니다."},
      {"start": 68.64,"end": 70.78, "raw_text": "교체보다 이걸 먼저 해줘야 합니다."},
      {"start": 90.0, "end": 99.62, "raw_text": "클릭 소리가 두번이 나게 끼워야 합니다. 안그러면 통전이 되지 않습니다."}
    ] }
  ```
  - 참고: 이 파일의 `normalized_text`는 초기엔 `raw_text`와 같고(정규화 대상 단어가 이 클립엔 없음), 정제(반복 태깅·정규화)는 메모리상에서 융합 직전에 적용된다. "아멘"은 무음 구간 오인식으로 보이며 환각 플래그는 false(연속 반복이 아니라 단발이라 현재 규칙으로는 안 잡힘) — **확인된 한계**.

## 3. 비디오 파이프라인 상세

- **YOLO** (물체 검출 모델, You Only Look Once): 튜닝된 `.pt` 가중치 + `ultralytics` 라이브러리. 역할 = 프레임에서 부품/손 위치(bounding box, 네모 박스)를 찾음. 실제 학습된 7클래스(`hand`, `RAM`, `RAM_slot`, `eraser`, `power_button_LED`, `GPU`, `monitor` 등 — `motherboard`는 의도적으로 학습 제외).
- **VLM** (영상-언어 모델, Vision-Language Model): **Qwen3-VL-8B-Instruct**, 4bit NF4 양자화(메모리 절약용 압축). 역할 = 영상을 보고 **관찰 가능한 행동만** 글로 적음(해석·암묵지 판단 금지 — 그건 LLM 몫). "눈" 역할.
- **YOLO가 VLM에 넘기는 것** (실제 코드 확인): YOLO 검출 결과에서 **부품 클래스 이름만 뽑아 중복 제거** → VLM 프롬프트에 `[고정 사실] 이 영상에 등장하는 부품은 다음과 같다(이 식별을 신뢰하라): RAM, RAM_slot, …` 블록으로 주입(부품 오인식 방지). **위치(좌표)는 안 넘김 — 클래스 이름만.** (스펙엔 "위치 힌트"라 적혔으나 구현은 식별만 — 결정 대기 항목.)
- **입력 → 처리 → 출력**:
  - 입력: ffmpeg로 뽑은 프레임들(YOLO·VLM 공용, 1회 추출) + (있으면)YOLO 부품 목록.
  - 처리: VLM이 프레임을 네이티브 비디오로 보고 관찰 로그(JSON) 생성. 각 관찰에 시각을 붙이되, **모델이 찍은 시각을 실제 프레임 그리드 시각에 스냅(교정)**함(오늘 추가한 타임스탬프 교정 — 상세 `_실행런타임_보고서.md` §4).
  - 출력: 관찰 리스트(`ActionDescription`: timestamp, actor, action, objects_visible).
- **이번 런 VLM 원시 관찰 산출물**: **디스크에 안 남음.** ⚠️
  - 이유: orchestrator는 VLM 관찰을 메모리로만 들고 다음 단계(정렬→LLM)로 넘기고, **최종 tacit JSON만** 파일로 저장한다. 중간 관찰 로그를 저장하는 코드가 없다.
  - 남기려면: `orchestrator.py`에서 `actions`(VLM 결과)를 받은 직후 `json.dump`로 예: `output/<video_id>.observations.json`에 저장하는 줄을 추가하면 됨. (현재는 미구현.)
  - **이번 런에 YOLO=noop(생략)이었으므로 부품 주입도 없었고**(검증용 설정), VLM은 자체 인식으로 관찰함.

### 3.1 YOLO 검출률 실측 — 4클립 (2026-06-30, fps 2.0, 검출만 단독 실행)

갭 B(YOLO 백업 그물) 설계 전에 "YOLO가 hand·부품을 실제로 얼마나 잡나"를 4클립 전부 측정. (frame% = 그 클래스가 1개 이상 나온 프레임 비율, conf=평균 신뢰도)

| 클래스 | CLIP1(조립) | CLIP2 | CLIP3(세척) | CLIP4(BIOS화면) |
|---|---|---|---|---|
| **hand** | **72%** (0.78) | **81%** (0.74) | **93%** (0.80) | **5%** (0.57) |
| RAM | 8% | 17% | **89%** | — |
| RAM_slot | 5% | 0% | 0% | 0% |
| eraser | — | — | **35%** | — |
| GPU | 7% | 63% | 23% | — |
| power_button_LED | 5% | 24% | 12% | 12% |
| monitor | 9% | — | — | **74%** |

- 속도: 4클립 합쳐 ~30초(31ms/프레임). **본선 고fps(5fps+)도 비용 무시 가능.**
- **hand**: 손 작업 클립(1·2·3)에서 72~93%로 잘 잡힘. **CLIP4만 5%** — BIOS 화면 보는 모드라 손이 화면 밖(monitor 74%). 놓친 게 아니라 손 작업 자체가 없는 구간.
- **RAM_slot**: 전 클립 ≤5%로 일관되게 **거의 못 잡음**(삽입 중 손/RAM에 가려짐). "손이 슬롯 접촉" 이벤트는 기대 어려움.
- **eraser**: CLIP3(세척)에서만 35% — 내용 의존이지만 있을 땐 잡음.
- **갭 B 함의**: 접촉을 `hand ∩ 특정부품`으로 좁히면 RAM_slot 약점에 걸림 → **1차 규칙은 "hand 검출 O + 같은 시점 VLM 관찰 X → action_only 후보"**(부품 무관), 부품 겹침(RAM/GPU/LED/eraser)은 보조 힌트로. (확정 전 검토 항목.)

## 4. 융합(LLM) 단계

- **모델**: **Qwen2.5-7B-Instruct-AWQ**, **vLLM** 0.23 추론 엔진으로 구동(AWQ = 4bit 양자화, vLLM = 빠른 LLM 서빙 엔진). (스펙 기본은 14B지만 8GB GPU 제약으로 7B로 다운그레이드.)
- **합쳐지는 방식**:
  1. 정렬 단계가 VLM 관찰 + 발화를 **±4초 윈도우(구간)**로 묶음(행동과 말은 동시에 안 일어나니 1:1 매칭 대신 구간으로).
  2. LLM이 각 구간을 보고 케이스 분류(A=행동+발화, B=행동만/action_only, C=발화만) → situation/tacit_insight/reasoning/diagnostic_steps를 채움.
  3. LLM은 LED 진단표(LLM만 봄)·근거성 판단을 직접 하고, 추론은 `model_inferred`로 정직 태깅.
  4. 결과를 스키마(v1.3)로 검증 → `output/<video_id>.tacit.json` 저장.
- **이번에 나온 최종 후보 JSON** (경로: `output/CLIP3_재부팅시도 전까지.mp4.tacit.json`, 후보 1건):
  ```json
  {
    "schema_version": "1.3",
    "video_id": "CLIP3_재부팅시도 전까지.mp4",
    "candidates": [{
      "id": "CLIP3_재부팅시도 전까지.mp4",
      "metadata": {
        "scenario_id": "...", "equipment": null, "task": null, "keywords": [],
        "scenario_title": null,
        "source": {"video_id": "CLIP3_재부팅시도 전까지.mp4",
                   "clip_start": "00:00:00", "clip_end": "00:01:39", "transcript_ref": null}
      },
      "knowledge": {
        "conflict": false, "conflict_detail": null,
        "situation": " Technician is performing a RAM module insertion and removal process, followed by a power button test and cleaning advice.",
        "situation_source": ["00:00:00","00:01:39"],
        "tacit_insight": " Technician is ensuring proper insertion and removal of RAM modules, and checking the power button LED.",
        "reasoning": " Technician is following a sequence of actions to insert and remove RAM modules, and testing the power button. The cleaning advice suggests the importance of maintaining the cleanliness of the components.",
        "reasoning_source": ["00:00:00","00:01:39"],
        "reasoning_origin": "utterance",
        "diagnostic_steps": [
          {"order": 1, "action": "Inserting and removing RAM modules",
           "evidence": "utterance", "source_utterance": "시작합니다", "timestamp": "00:00:00"},
          {"order": 2, "action": "Cleaning advice",
           "evidence": "utterance", "source_utterance": "깨끗해 보여도 이렇게 한 번씩 닦아줘야 합니다.", "timestamp": "00:01:00"},
          {"order": 3, "action": "Ensuring proper contact",
           "evidence": "action_only", "source_utterance": null, "timestamp": "00:00:48"},
          {"order": 4, "action": "Testing power button",
           "evidence": "action_only", "source_utterance": null, "timestamp": "00:00:20"},
          {"order": 5, "action": "Ensuring proper contact",
           "evidence": "action_only", "source_utterance": null, "timestamp": "00:00:52"},
          {"order": 6, "action": "Cleaning advice",
           "evidence": "utterance", "source_utterance": "클릭 소리가 두번이 나게 끼워야 합니다. 안그러면 통전이 되지 않습니다.", "timestamp": "00:01:30"}
        ]
      }
    }]
  }
  ```
  - ✅ **갱신됨(2026-06-30, 갭 A 반영 런)**: `knowledge`에 **`"conflict": false, "conflict_detail": null` 정상 출력 확인**. CLIP3엔 VLM(본 것)/STT(들은 것) 충돌이 없어 `false`가 맞음 — 충돌 보존 배관이 end-to-end로 동작함을 확인. (실제 `true`는 LED 횟수 불일치 같은 충돌 데이터에서 발현.)
  - 참고: LLM은 비결정적이라 재실행 시 step 구성·문구가 달라진다(이번엔 6스텝). `scenario_id`가 리터럴 `"..."`로 나온 건 **신규 품질 이슈**(LLM이 채우지 못함).
  - **관찰된 품질 이슈**(타임스탬프와 무관, 후속 과제): ① situation/reasoning이 **영어로** 출력됨(한국어 프롬프트인데). ② 발화 내용(단자 세척·산화막 등 노하우)이 후보에 거의 반영 안 되고 LED 관찰만 반복됨 — 융합 품질은 별도 튜닝 필요. ③ `transcript_ref`가 null(STT 파일 경로 연결 미설정).

## 5. 모델/환경 사양 정리표

> 환경변수·디바이스 배치·OOM 함정 등 상세는 `_실행런타임_보고서.md` §1~3 참조. 여기선 단계별 모델/설정 요약만.

| 단계 | 모델/도구 | 버전·식별자 | 디바이스(이번 검증 런) | 주요 설정 |
|---|---|---|---|---|
| STT | Whisper large-v3-turbo (faster-whisper) | `mobiuslabsgmbh/faster-whisper-large-v3-turbo` | **CPU**(검증) ↔ 본선 cuda | ko, `condition_on_previous_text=False`, 검증 `compute_type=int8` ↔ 본선 float16 |
| 프레임추출 | ffmpeg CLI | `/home/piai/anaconda3/envs/deep/bin/ffmpeg` | CPU | fps 0.25(검증) ↔ 본선 0.5, long_side 480 |
| 검출 | YOLO (ultralytics) | 튜닝 `best.pt`(dell7920_v1) | **검증=noop(생략)** ↔ 본선 cuda | conf 0.25, imgsz 640, 7클래스 |
| VLM | Qwen3-VL-8B-Instruct | NF4 4bit, transformers `sdpa` | **cuda:0(빈 카드)** | 검증 max_pixels 160²·max_new_tokens 1024 ↔ 본선 192²·2048(스펙 4000) |
| 정렬 | WindowAligner (코드) | — | CPU | window ±4초 |
| 융합 LLM | Qwen2.5-7B-Instruct-AWQ | vLLM 0.23, AWQ 4bit | cuda:0(VLM unload 후) | `TRITON_ATTN`, `enforce_eager`, dtype half, max_model_len 4096, temp 0.2 / 본선 후보=14B(2GPU TP2) |

**검증용으로 낮춘 값 ↔ 본선 값 핵심:**
- detector: `noop`(검증) ↔ `yolo_ultralytics`(본선)
- STT: CPU/int8(검증) ↔ GPU/float16(본선)
- VLM: 160²·fps0.25·1024토큰(검증, OOM 회피) ↔ 192²·fps0.5·2048+토큰(본선, 단 빈 GPU 확보 전제)
- LLM: 7B-AWQ(현재 8GB) ↔ 14B-AWQ(서버/2GPU)
