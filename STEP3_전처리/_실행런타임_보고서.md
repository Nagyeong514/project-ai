## 2. 환경 함정 6종 + 해결 (이번 세션 실측)

### 함정 1 — ultralytics(YOLO)가 `CUDA_VISIBLE_DEVICES`를 덮어쓴다 ⭐
ultralytics `select_device()`는 device 인자에 따라 프로세스의 `CUDA_VISIBLE_DEVICES`를 **강제로 재설정**한다:
`device="cuda:0"→"0"`, `"cuda:1"→"1"`, `"cpu"→"-1"`.
→ YOLO 단계([3/5]) 이후 같은 프로세스의 VLM이 `device_map="cuda:1"`을 쓰면 **`invalid device ordinal`**, YOLO를 cpu로 두면 **`No CUDA GPUs are available`**로 죽는다. 디바이스 배치가 계속 꼬인 근원.
- **해결(검증 런):** `detector.impl: noop` — GPU/모델을 일절 안 건드리는 신규 `NoopDetector` 추가(`components/detector_noop.py`, registry 1줄). 타임스탬프 검증엔 YOLO 불필요.
- **본선 권고:** YOLO와 VLM을 **같은 가시 디바이스**에 두거나(공존 메모리 주의), YOLO를 **별도 서브프로세스**로 분리.

### 함정 2 — 8GB VLM 생성 OOM
VLM 로드는 통과해도 **생성(prefill SDPA attention)**에서 활성 메모리가 터진다. 빈 카드 아니면 74–78MB 차이로 부족.
- **실측:** busy 카드(GPU1, 480MB 점유) → OOM / 빈 카드(GPU0, 7728MB free) → 6250MB로 통과(900MB+ 마진).
- **해결:** ① VLM을 빈 카드로 ② 설정 하향(3절 표). `max_pixels`보다 **카드 선택**이 결정적이었다(하향만으론 7.06GB로 여전히 부족).

### 함정 3 — STT가 GPU를 물고 안 놓는다
`WhisperTurboSTT.transcribe()`는 **캐시(transcripts/*.json)를 재사용하지 않고 매번 모델을 GPU에 로드**한다. orchestrator는 VLM 전에 STT/YOLO를 unload하지 않아, 첫 런에서 cuda:0에 2.38GB(whisper+YOLO)가 남은 채 VLM 로드 → OOM.
faster-whisper(ctranslate2)는 **`"cuda:1"` 표기를 거부**(device="cuda" + device_index 분리 필요)라 GPU1로 쉽게 못 보냄.
- **해결(검증 런):** `stt.device: cpu`, `compute_type: int8`. CLIP3(111s)는 CPU로도 수 분 내.
- **본선 권고:** STT를 다른 GPU로 보내려면 컴포넌트에 `device_index` 노출 필요(현재 미노출).

### 함정 4 — vLLM 워커 MKL 스레딩 충돌
VLM이 CUDA를 먼저 초기화해서 vLLM이 워커를 **spawn**으로 띄우는데, 그 자식 프로세스가
`MKL_THREADING_LAYER=INTEL is incompatible with libgomp.so.1`로 죽어 **`Engine core initialization failed`**.
- **해결:** `MKL_THREADING_LAYER=GNU`. (이번에 처음 [5/5]까지 도달해 드러난 함정.)

### 함정 5 — Turing(sm75) 전반 제약
bf16·FP8·FlashAttention2·FlashInfer 전부 불가.
- **VLM:** `attn_implementation="sdpa"`(이미 코드에 적용, FA2 불가), BitsAndBytes NF4(compute fp16).
- **LLM(vLLM):** `attention_backend="TRITON_ATTN"`, `enforce_eager=True`, `dtype="half"`, `quantization="awq"`, `VLLM_USE_FLASHINFER_SAMPLER=0`, `max_num_seqs=16`.
- `LD_LIBRARY_PATH=/home/piai/anaconda3/lib` 선행 필수.

### 함정 6 — 영상 디코딩 / 한글 경로
torchcodec·pyav·cv2 이 env서 불안정 → **ffmpeg CLI subprocess로 jpg 1회 추출**(YOLO·VLM 공용). 원본 `명장님비디오/` 디렉토리명 인코딩 깨짐 → **`master/master_videos/`(ASCII)** 사용.

## 3. config 튜닝 표 (본선 기본 ↔ 검증용 하향)

| 항목 | 본선 기본(스펙) | 검증용 하향(현재 config) | 이유 |
|---|---|---|---|
| `stt.device` | `cuda` | `cpu` | GPU 경쟁 회피(함정 3) |
| `stt.compute_type` | `float16` | `int8` | cpu 호환 |
| `detector.impl` | `yolo_ultralytics` | `noop` | CVD 장악 회피(함정 1) |
| `vlm.device` | `cuda:0` | `cuda:0` (빈 카드) | OOM 회피(함정 2) |
| `vlm.max_pixels` | `36864` (192²) | `25600` (160²) | OOM 마진 |
| `vlm.fps_short` | `0.5` | `0.25` (≈28프레임) | 프레임수↓ → 활성 메모리↓ |
| `vlm.max_new_tokens` | `2048` (스펙 4000) | `1024` | 생성 KV캐시↓ |
| `llm.model_name` | `Qwen2.5-7B-AWQ` | 동일 | 14B는 2GPU TP2 프리셋만 |

### 본선 원복 가이드
1. `detector.impl → yolo_ultralytics` (단, 함정 1 재발 — YOLO/VLM 디바이스 공존 전략 필요).
2. `stt.device → cuda` (단, VLM 전 unload 또는 별 GPU 필요).
3. VLM 파라미터 상향(`max_pixels 192²`, `fps 0.5`, `tokens 2048+`) — **단 빈 카드 확보 전제**. 8GB busy 카드면 OOM 재발.
4. 서버(더 큰 GPU) 이관 시 양자화 해제·14B(TP2)로 교체 — 전부 config로.

> ⚠️ 검증용 하향값은 **OOM을 피해 파이프라인을 한 번 끝까지 돌리기 위한 것**이지, 관찰 품질 최적값이 아니다. 품질 실험은 빈 카드/큰 GPU에서 기본값으로.

## 4. VLM 타임스탬프 교정 (이번 세션 수정 + 검증)

### 4.1 배경
VLM(Qwen3-VL)이 출력 텍스트에 timestamp를 **직접 찍는데**, 모델은 실제 비디오 메타데이터(진짜 fps)를 못 받고 `fps=24`로 가정한다(로그 경고 실측). 그래서 모델 추정 시각은 못 미덥다. **우리는 실제 프레임 그리드 시각(`times = i / 샘플링fps`)을 손에 쥐고 있으므로** 그 값으로 교정한다 — "시간은 우리가 부여한 걸 들고 간다"는 원칙(설계 5.6) 복원.

### 4.2 수정 3종 (`components/vlm_qwen.py`)
1. **그리드 스냅** `snap_to_grid(ts, grid)`: 모델 출력 ts를 `times` 중 가장 가까운 값으로 교정.
2. **파싱 폴백** `_even_grid_ts(idx, total, grid)`: timestamp 누락/파싱실패 시 **0:00 떼몰림 대신** 그리드를 관찰 순서대로 균등 배분.
3. **순서 검증**: 반환 직전 timestamp 단조증가로 안정 정렬, 순서 바뀐 관찰 있으면 `logger.warning("[vlm] reordered N observations")` (내용은 안 버림).

(주의: `orchestrator._probe_meta`의 `fps=30.0` 하드코딩은 `observe_video`/motion 경로에서만 쓰이며 본선 `observe_frames` 경로엔 무관 — `# TODO(decision)` 주석만 달아둠.)

### 4.3 검증 결과 (CLIP3, fps0.25 → 그리드 0,4,8,…초) — ✅ 통과 (2026-06-30)
- **end-to-end 실행:** ✅ exit 0. 후보 1건 저장(`output/CLIP3_….mp4.tacit.json`), 스키마 v1.3 정상.
- **순수 단위 검증(GPU 불필요):**
  - `snap_to_grid`: `3.2→4.0`, `9.9→8.0`, `17.0→16.0`, `23.7→24.0`, **`99.0→24.0`(범위 밖 클램프)**, 빈 그리드→원값 통과. ✅
  - `_even_grid_ts`: n=4 → `[0,8,16,24]` **균등배분**(0:00 떼몰림 없음), n=1→`[0]`, 빈 그리드→`0.0`. ✅
- **`[vlm] reordered` 경고:** 미발생 = VLM이 단조증가 시각을 출력(순서검증이 정상 통과, 헛경고 없음).
- **출력 JSON timestamp 정합성:**
  - `action_only` step(VLM 유래) `00:00:20` = **그리드 배수(5×4) → 정렬 확인** ✅.
  - 발화 기반 step(`00:00:00`, `00:01:30`)은 STT 시각 사용 — 설계 5.6대로 정상(STT는 자체 시각).
- **한계(정직 기록):** 최종 JSON은 LLM이 생성하고 VLM 원시 observations는 디스크에 저장되지 않으므로, JSON만으론 스냅을 직접 증명 불가 → **순수 단위 검증으로 함수 동작을 확정**했고, JSON의 action_only 시각이 그리드에 맞는 것으로 교차 확인.
