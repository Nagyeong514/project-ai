## 1. 문제 분류 (당황하지 말 것 — 대부분 A그룹이다)

### A. 🟢 서버/큰 GPU 받으면 그냥 사라지는 것 — 코드 잘못 아님, 지금 손대지 마라
오늘 무서워 보인 에러 대부분이 여기 속한다. **8GB Turing 한 장**이라는 악조건의 부산물일 뿐, 파이프라인 코드는 이 조건에서도 end-to-end로 돌았다.

| 문제 | 원인 | 서버에서 |
|---|---|---|
| VLM 로드/생성 OOM | 8GB에 6.9GB 모델 빠듯 | 큰 GPU면 소멸. 검증용 하향값 전부 원복 |
| 디바이스 꼬임(cuda:0↔1) | 카드 2장·타프로세스 점유 | GPU 넉넉하면 곡예 불필요 |
| ultralytics가 CUDA_VISIBLE_DEVICES 덮어씀 | YOLO·VLM 한 프로세스 공존 | 여유 GPU면 noop 우회 불필요 |
| VLM 출력 토큰 잘림 | `max_new_tokens` 1024로 낮춤(OOM 회피) | 4000으로 원복 |
| LLM 컨텍스트 4096 초과 | KV캐시 빠듯 | `max_model_len` 키울 여유 |

→ 결론: **A그룹은 "환경 교체로 해결". 지금 8GB에 맞춰 잡아봤자 내일 다시 풀어야 함.**

### B. 🟡 큰 모델로 바꾸면 "좋아질" 것 — 서버 후 다시 측정하고 남는 것만 손봄
| 문제 | 큰 모델이면 | 그래도 병행 필요 |
|---|---|---|
| VLM 반복환각 루프(`modeling_table` 등 없는 객체 지어냄) | 줄지만 완전 보장 X | 부품주입 ON + 프롬프트(§3) |
| LLM이 관찰을 발화로 둔갑 / 실제 발화 누락 | 14B+면 지시 따르기↑ | 프롬프트 sharpen(§3) |
| 출력이 영어로 나옴 | 큰 모델이면 한국어 잘 따름 | 프롬프트에 "한국어" 명시 |

→ **B그룹은 "서버에서 좋은 모델로 재측정 → 남는 것만 프롬프트". 지금 8B/7B로 튜닝해도 모델 바뀌면 무의미.**

### C. 🔴 모델 바뀌어도 안 사라짐 — 코드·프롬프트로 잡아야 (모델 무관)
| 문제 | 성격 | 상태 |
|---|---|---|
| STT 환각 상투구("다음영상에서 만나요/감사합니다/아멘") | Whisper 고유 버릇, 체급 무관 | ✅ **denylist 태깅으로 오늘 해결** |
| STT 희소·라운드 타임스탬프 | Whisper 침묵 처리 특성 | VAD는 병합 부작용 → OFF 유지 + denylist 선택 |
| LLM 융합이 관찰↔발화 채널 혼동 | 프롬프트 명세 부족 | ⬜ 프롬프트 과제(§3) |
| `cross_check` 빈틈(가짜 발화 통과) | 검증 규칙 허술 | ⬜ 코드 보강(§3) |
| metadata 미충전(`scenario_id="..."`, `transcript_ref=null`, task/equipment null) | wiring·프롬프트 갭 | ⬜ 코드/프롬프트 |
| VLM 객체 환각 | 부품주입 OFF(noop) | ⬜ YOLO 주입 ON(서버 무관하게 도움) |

→ **C그룹이 진짜 우리 숙제.** 양은 적고 명확하다.

## 2. 본선 전환 체크리스트 (config 원복 + 확인)

> 값/함정 근거는 `_실행런타임_보고서.md` §3. 여기선 실행 체크리스트만.

**config.yaml 원복:**
- [ ] `detector.impl`: `noop` → `yolo_ultralytics`
- [ ] `stt.device`: `cpu` → `cuda` (별 GPU면 device_index), `compute_type`: `int8` → `float16`
- [ ] `vlm.device`: 검증용 카드지정 → GPU 넉넉하면 단순화
- [ ] `vlm.max_pixels`: `25600`(160²) → `36864`(192², 스펙)
- [ ] `vlm.fps_short`: `0.25` → `0.5` (서버 성능 충분하면 1~2)
- [ ] `vlm.max_new_tokens`: `2048` → `4000`(스펙)
- [ ] `llm.model_name`: `7B-AWQ` → `14B-AWQ`/더 큰 모델, `tensor_parallel_size` 조정
- [ ] `llm.max_model_len`: `6144` → 관찰 규모 보고 상향(관찰 많으면 8K+), `max_new_tokens` 본선값

**유지/재검토 환경:**
- [ ] 유지: `MKL_THREADING_LAYER=GNU`, `LD_LIBRARY_PATH=…/anaconda3/lib`
- [ ] **재검토**: 서버 GPU가 sm80+ (Ampere↑)면 Turing 우회 불필요 → `TRITON_ATTN`/`enforce_eager`/`VLLM_USE_FLASHINFER_SAMPLER=0` 해제 가능, VLM도 FlashAttention2 켜기, bf16 사용 가능
- [ ] `orchestrator._probe_meta`의 `fps=30.0` 하드코딩 — `observe_video`/motion 경로 쓸 거면 실제 fps로 교체

**전환 후 재측정:**
- [ ] 본선 설정으로 CLIP1~4 전부 돌려 **VLM 반복환각·객체환각이 큰 모델에서 줄었는지 재평가**(B그룹)
- [ ] 남은 환각/채널혼동만 §3 프롬프트로 마무리

## 3. 프롬프트 개선 과제 (모델 바뀌어도 챙길 것 — 프롬프트/검증 전용)

### 3.1 VLM 관찰 프롬프트 (`prompts/vlm_observation.py`)
- **반복 억제**: "직전과 동일한 행동/객체를 반복 나열하지 마라. 새로 관찰된 것이 없으면 출력을 멈춰라." (8B가 36초 이후 4스텝 루프를 돈 실측 있음)
- **객체 환각 억제**: "objects_visible에는 **주입된 부품명** 또는 화면에 명확히 보이는 일반 명사만. 추측 객체명 금지." (`modeling_table`, `card_component`, `planner` 등 없는 객체 생성 실측) — 부품주입 ON과 병행.
- **출력 언어**: 한국어 고정(이미 한국어지만 명시 유지).

### 3.2 LLM 융합 프롬프트 (`prompts/llm_fusion_prompt.py`)
- ⭐ **채널 분리 못박기**(가장 중요): "`source_utterance`는 **utterances(STT 발화) 원문 전용**이다. **observations(영상 관찰) 문장을 source_utterance에 절대 넣지 마라.** 관찰만 있고 발화 없으면 evidence=action_only, source_utterance=null." — 실측서 LLM이 VLM 관찰("전원 버튼 LED가 주황색으로…")을 발화인 척 넣고 evidence=utterance로 태깅한 사고 있음.
- **실제 발화 누락 방지**: "제공된 utterances(repeat_hallucination=true 제외)는 빠짐없이 후보 생성에 반영하라." — 실측서 진짜 STT 노하우("닦아야/산화막/클릭 두 번")가 통째로 누락됨.
- **출력 언어**: "모든 문자열 필드는 한국어로 작성." — 영어 출력 실측.

> **⚠️ 2026-07-01 결과 — 위 프롬프트 하드닝(#2)을 7B에 실제 적용·재측정:**
> - **출력 언어 = ✅ 해결**(7B도 한국어로 나옴, situation/reasoning/action 전부 한글).
> - **발화 완전반영 = ❌ 7B가 여전히 무시.** "빠짐없이 반영"을 프롬프트에 못박았는데도 CLIP3 6개 발화(닦아야/산화막/교체먼저/클릭두번)를 **전부 누락**하고, 그 자리에 **VLM 관찰문을 발화(source_utterance)로 둔갑**시켜 채움(cross_check가 step2·4로 자동 탐지).
> - **결론: 발화 누락·채널 둔갑은 프롬프트로 더 못 누른다 = 7B capacity 한계(B그룹 확정).** → **서버 14B+에서 동일 프롬프트로 재측정**해서 실제로 녹는지 확인이 다음 관문. 안 녹으면 그때 입력 재배치(발화 우선/강조) 등 프롬프트 2차 시도.

### 3.3 검증 코드 보강 (`schema/tacit_schema.py` cross_check) — ✅ 2026-07-01 완료
- ~~현재 cross_check는 "evidence=utterance인데 source_utterance 빈 경우"만 잡고, 가짜 발화(관찰을 넣은 것)는 통과시킴.~~
- **구현됨**: `cross_check(observation_texts, utterance_texts)`로 확장. ① source_utterance가 VLM 관찰문과 일치 → "관찰을 발화로 둔갑 의심", ② source_utterance가 입력 발화에 없음 → "발화 지어냄 의심" 경고. orchestrator가 VLM 관찰·STT 발화를 넘김(짧은 문자열 8자 하한 오탐방지).
- **실전 검증**: CLIP3 재측정서 7B의 관찰 둔갑(step2·4)을 실제로 자동 탐지. **이제 cross_check 경고 = "융합 실패 신호"로 활용** 가능.

## 한 줄 정리
오늘 에러 폭탄의 **8할은 8GB 환경 부산물(A)** — 서버 오면 녹는다. **모델 체급 이슈(B)**는 서버 후 재측정. **진짜 우리 숙제(C)**는 STT denylist(완료) + 융합 프롬프트 채널분리 + cross_check 보강 정도로 적고 명확하다.
